# 轻量化零信任访问代理与审计网关开发设计需求文档

## 1. 项目背景与建设目标

### 1.1 背景

公司计划自研一套面向 `e-cology / OA` 的轻量化零信任访问代理与审计网关，用于逐步替代现有外部产品在 OA 入口保护、访问审计和灰度接入中的能力。

一期建设不追求完整零信任平台，而是优先解决 OA 统一入口的认证承接、入口授权、访问审计、轻量管理和快速回滚问题。系统以 Web 七层反向代理为核心形态，部署在 OA 入口前方，通过网关鉴权决定用户是否可以进入 OA 应用。

### 1.2 建设目标

- 建设一套可落地的一期零信任入口网关，优先接入 `e-cology / OA`。
- 由 OA 自身完成用户账号密码认证，零信任通过 OA 登录成功回调建立独立会话。
- 对 OA 入口做应用级授权控制，不改造 OA 内部审批、流程、报表、人事、财务等模块权限。
- 采集完整访问元数据，并支持按规则留存脱敏后的请求体和响应体，支撑安全排查、访问溯源和灰度观察。
- 提供轻量管理后台，支持应用、用户、用户组、授权策略和日志查询。
- 采用单机旁路部署和小范围 DNS/hosts 灰度方式，保留原 OA 链路，支持快速回滚。

## 2. 一期范围与非目标

### 2.1 一期范围

- Web 七层访问代理：HTTP/HTTPS 反向代理、Header 透传、Cookie 保持、上传下载转发、超时控制。
- OA 登录承接：未登录用户跳转 OA 登录页，OA 登录成功后回调零信任。
- 零信任会话：回调验证成功后创建 `ztrust_session`，后续请求通过零信任会话鉴权。
- 入口授权：按 OA 应用、用户、用户组配置允许访问策略。
- 审计日志：记录访问日志、登录/回调日志、管理操作日志。
- 管理后台：支持应用管理、用户映射、用户组、授权策略、日志查询。
- 部署验证：先接 OA 测试环境，再做小范围用户灰度。

### 2.2 非目标

- 不做四层 VPN，不接入 SSH、RDP、数据库等非 Web 资源。
- 不做终端 Agent、设备基线采集、设备指纹强绑定。
- 不做 DLP 和动态内容识别；请求体和响应体仅按显式规则留存，保存前必须脱敏。
- 不做 OA 内部模块权限重构。
- 不做动态风险评分、持续自适应访问控制。
- 不做双机高可用和统一日志平台，一期仅为后续演进预留接口和数据结构。
- 不以完全隐藏 OA 登录口为一期目标，OA 自身安全加固仍需保留。

## 3. 总体架构

### 3.1 技术路线

- 网关层：OpenResty / Nginx
- 控制面与管理后台：Go
- 数据库：MySQL
- 缓存与会话：一期默认使用 MySQL，Redis 作为二期增强选项
- 部署模式：单机旁路，支持灰度和快速回滚
- 接入对象：一期仅接入 `e-cology / OA`
- 认证方式：OA 自身认证 + OA 回调零信任

### 3.2 逻辑架构

```text
用户浏览器
  ↓
OA 访问域名
  ↓
OpenResty / Nginx 零信任网关
  ↓
Go 控制面 / 鉴权服务
  ↓
MySQL: 应用、用户映射、用户组、授权策略、ZT 会话、ticket、nonce、审计日志
  ↓
OA 后端服务
```

### 3.3 核心原则

- OA 负责“用户是谁”的账号密码认证。
- 零信任负责“该用户是否允许进入 OA 入口”的入口级鉴权。
- OA 内部业务权限继续由 OA 自身控制。
- 零信任保存独立会话，避免每个请求都依赖 OA 在线校验。
- 日志默认记录访问元数据；请求体和响应体必须命中正文审计规则后才保存，且只保存脱敏后的内容。

## 4. 访问与认证流程

### 4.1 正常访问流程

1. 用户访问 OA 域名。
2. OpenResty 对请求发起内部鉴权请求 `GET /auth/check`。
3. 鉴权服务检查浏览器是否携带有效 `ztrust_session`。
4. 如果会话有效，继续检查该用户是否有 OA 应用入口访问授权。
5. 鉴权通过后网关代理请求到 OA 后端。
6. 网关在请求结束后上报访问日志到 `POST /audit/access`。

### 4.2 未登录流程

1. 用户访问 OA 域名但没有有效 `ztrust_session`。
2. 网关收到 `GET /auth/check` 的 401 结果。
3. 用户被引导到 `GET /auth/oa/start`。
4. 零信任生成登录 state，并将用户重定向到 OA 登录页。
5. OA 完成自身账号密码认证。
6. OA 登录成功后回调零信任 `GET /auth/oa/callback`。
7. 零信任验证回调参数，成功后创建 `ztrust_session` 并跳回原始访问地址。

### 4.3 OA 回调校验流程

OA 回调零信任时携带以下核心参数：

- `ticket`：OA 生成的一次性登录票据。
- `oa_account`：OA 登录账号，作为零信任用户唯一标识。
- `timestamp`：回调生成时间。
- `nonce`：随机数，用于防重放。
- `signature`：签名值。
- `return_to`：登录完成后的回跳地址。

零信任侧校验要求：

- `timestamp` 必须在允许时间窗口内。
- `nonce` 必须未使用过，校验后写入 MySQL 并设置过期时间。
- `ticket` 必须存在、未过期、未使用，校验后立即作废。
- `signature` 必须匹配约定签名算法。
- `return_to` 必须限制在允许的 OA 域名或相对路径范围内，避免开放跳转。

签名建议：

```text
signature = HMAC-SHA256(secret, ticket + "\n" + oa_account + "\n" + timestamp + "\n" + nonce + "\n" + return_to)
```

一期可以根据 OA 改造能力选择由 OA 回源校验 ticket，或由零信任本地校验 OA 写入 MySQL 的 ticket。默认推荐“OA 生成 ticket + 零信任回源或共享密钥验签 + MySQL 标记作废”。

### 4.4 退出流程

1. 用户点击退出或访问 `POST /auth/logout`。
2. 零信任删除或失效 MySQL 中的 `ztrust_session`。
3. 零信任清理浏览器 Cookie。
4. 是否同步退出 OA 会话由 OA 能力决定；一期默认不强制接管 OA 内部退出。

## 5. 核心功能需求

### 5.1 访问网关

- 支持 OA 域名七层反向代理。
- 支持 HTTP/HTTPS 访问，生产应启用 HTTPS。
- 支持 Header 透传，包括 `Host`、`X-Real-IP`、`X-Forwarded-For`、`X-Forwarded-Proto`。
- 支持 Cookie 和 Session 保持，不破坏 OA 原有登录态。
- 支持附件上传下载和大文件请求。
- 支持基础超时控制、上传大小限制和代理错误记录。
- 支持网关访问日志上报。

### 5.2 OA 登录承接

- 未登录访问时跳转 OA 登录页。
- OA 登录成功后回调零信任。
- 零信任校验回调参数并建立独立会话。
- 回调失败时展示明确拦截原因或错误页。
- 支持将用户回跳到原始访问路径。

### 5.3 授权控制

- 支持按应用配置入口访问授权。
- 支持按 OA 账号授权。
- 支持按用户组授权。
- 支持启用/停用应用。
- 未授权访问返回 403 或展示拦截页。
- 策略变更必须记录管理操作日志。

### 5.4 日志审计

访问日志至少记录：

- OA 账号
- 来源 IP
- 访问域名
- 请求路径
- 请求方法
- 响应状态码
- 代理结果
- 请求耗时
- User-Agent
- 浏览器和操作系统
- 访问时间

登录/回调日志至少记录：

- OA 账号
- 来源 IP
- 结果
- 失败原因
- ticket 校验结果
- 时间

管理操作日志至少记录：

- 管理员账号
- 操作对象
- 操作类型
- 操作前后摘要
- 操作时间

日志默认保留 90 天。

### 5.5 管理后台

一期仅支持单一超级管理员模型，后台功能包括：

- 应用管理：应用名称、访问域名、后端地址、启停状态、代理参数。
- 用户映射：OA 账号、姓名、状态、最近登录时间。
- 用户组管理：组名、成员关系。
- 授权策略：应用、用户或用户组、允许访问、有效期。
- 访问日志查询：按账号、IP、路径、状态码、时间范围查询。
- 登录日志查询：按账号、IP、结果、时间范围查询。
- 管理操作日志查询。

## 6. 模块设计

### 6.1 网关模块

职责：

- 接收用户浏览器请求。
- 调用内部鉴权接口。
- 根据鉴权结果放行、跳转登录或拦截。
- 将放行请求转发到 OA 后端。
- 在请求结束后异步或短超时上报访问日志。

关键要求：

- 鉴权接口仅内网访问，不暴露给公网。
- 审计上报失败不应阻断正常 OA 访问，但必须在网关错误日志中记录。
- 上传下载请求默认不记录请求体和响应体；如确需审计，必须配置专门规则、较小截断阈值和更短保留期。

### 6.2 鉴权服务模块

职责：

- 校验 `ztrust_session`。
- 根据请求域名匹配应用。
- 根据用户、用户组和应用授权策略判断是否放行。
- 返回供网关注入的用户信息 Header。

放行时返回：

- HTTP 204
- `X-ZTrust-User-ID`
- `X-ZTrust-Username`
- `X-ZTrust-App-ID`

失败时返回：

- 401：未登录或会话失效。
- 403：已登录但无授权。
- 404：请求域名未配置应用。

### 6.3 OA 回调认证模块

职责：

- 生成登录 state。
- 处理 OA 登录成功回调。
- 校验 ticket、timestamp、nonce、signature。
- 建立零信任会话。
- 写入登录/回调日志。

失败处理：

- ticket 不存在、过期、已使用：拒绝并记录失败原因。
- 签名错误：拒绝并记录安全事件。
- nonce 重放：拒绝并记录安全事件。
- 用户未被纳入零信任用户映射：可按配置自动创建待启用用户或拒绝；一期默认拒绝并提示管理员配置授权。

### 6.4 管理后台模块

职责：

- 提供应用、用户、用户组、策略的管理界面。
- 提供审计日志查询界面。
- 记录每次管理操作日志。

一期权限模型：

- 仅超级管理员可登录后台。
- 不区分审计员、运维员、安全管理员等角色。

### 6.5 存储模块

MySQL 存储一期全部持久数据和短期状态：

- 应用配置
- OA 用户映射
- 用户组与成员关系
- 授权策略
- ZT 会话
- OA 登录 ticket 状态
- nonce 防重放记录
- 登录失败计数
- 访问日志
- 登录日志
- 管理操作日志

轻量缓存策略：

- 一期不强依赖 Redis。
- `/auth/check` 默认直接查询 MySQL。
- 如灰度并发升高，可在 Go 进程内增加 30 秒到 2 分钟的短 TTL 授权缓存。
- 进程内缓存只作为性能优化，策略变更后应主动清理或更新缓存。
- Redis 作为二期高并发、高可用增强组件，不纳入一期必需部署。
- MySQL-only 方案适合单机和小范围灰度；当访问量扩大、双机高可用或跨实例会话共享成为刚需时，再迁移会话和授权缓存到 Redis。

## 7. 接口设计

### 7.1 `GET /auth/oa/start`

用途：

- 未登录用户访问 OA 时，生成登录 state 并跳转 OA 登录页。

请求参数：

- `return_to`：登录完成后的回跳地址。

处理逻辑：

- 校验 `return_to` 属于允许域名或相对路径。
- 生成 state，写入 MySQL，设置短有效期。
- 重定向到 OA 登录页，并附带 state 或回调地址。

### 7.2 `GET /auth/oa/callback`

用途：

- OA 登录成功后回调零信任，建立 ZT 会话。

请求参数：

- `ticket`
- `oa_account`
- `timestamp`
- `nonce`
- `signature`
- `return_to`

成功响应：

- 设置 `ztrust_session` Cookie。
- 重定向到 `return_to`。

失败响应：

- 返回登录失败页或拦截页。
- 写入登录/回调失败日志。

### 7.3 `GET /auth/check`

用途：

- OpenResty 内部鉴权接口。

请求来源：

- 仅网关内部调用。

请求信息：

- Cookie：`ztrust_session`
- Header：`X-Original-Host`、`X-Original-URI`、`X-Real-IP`、`X-Forwarded-For`

响应：

- 204：允许访问。
- 401：未登录或会话失效。
- 403：无授权。
- 404：应用未配置。

### 7.4 `POST /auth/logout`

用途：

- 删除 ZT 会话并清理 Cookie。

处理逻辑：

- 从 Cookie 读取 session token。
- 删除或失效 MySQL session。
- 设置 Cookie 过期。

### 7.5 `POST /audit/access`

用途：

- 网关上报访问审计元数据，可选携带请求体和响应体。

请求体字段：

- `app_id`
- `user_id`
- `username`
- `source_ip`
- `domain`
- `path`
- `method`
- `status_code`
- `duration_ms`
- `proxy_result`
- `user_agent`
- `request_body.content_type`
- `request_body.body`
- `response_body.content_type`
- `response_body.body`

处理逻辑：

- 解析浏览器和操作系统。
- 根据正文审计规则判断是否保存请求体或响应体。
- 命中规则时对正文进行敏感字段脱敏、大小截断和哈希计算。
- 写入 MySQL 访问日志。
- 返回创建结果。

### 7.6 正文审计规则接口

用途：

- 管理请求体和响应体留存规则，默认不全量记录正文。

接口：

- `GET /admin/body-rules`
- `POST /admin/body-rules`
- `PUT /admin/body-rules/{id}`
- `DELETE /admin/body-rules/{id}`

规则字段：

- `name`
- `app_id`
- `path_pattern`
- `match_type`：`prefix`、`contains`、`exact`
- `methods`
- `status_min`
- `status_max`
- `capture_request`
- `capture_response`
- `max_body_bytes`
- `enabled`

### 7.7 访问日志详情接口

用途：

- 查看单条访问日志详情，包括已脱敏保存的请求体和响应体。

接口：

- `GET /admin/audit/access/{id}`

处理逻辑：

- 只有管理员可访问。
- 如果日志包含正文，查看动作必须写入管理操作日志。
- 列表接口只返回正文存在标识，详情接口才返回正文内容。

### 7.8 管理后台接口

应用管理：

- `GET /admin/apps`
- `POST /admin/apps`
- `PUT /admin/apps/{id}`
- `DELETE /admin/apps/{id}`

用户映射：

- `GET /admin/users`
- `POST /admin/users`
- `PUT /admin/users/{id}`
- `DELETE /admin/users/{id}`

用户组：

- `GET /admin/groups`
- `POST /admin/groups`
- `PUT /admin/groups/{id}`
- `DELETE /admin/groups/{id}`

授权策略：

- `GET /admin/policies`
- `POST /admin/policies`
- `DELETE /admin/policies/{id}`

审计查询：

- `GET /admin/audit/access`
- `GET /admin/audit/login`
- `GET /admin/audit/admin`

## 8. 数据与存储设计

### 8.1 应用

字段：

- 应用 ID
- 应用名称
- 访问域名
- 后端地址
- 启停状态
- 代理超时
- 创建时间
- 更新时间

### 8.2 OA 用户映射

字段：

- 用户 ID
- OA 账号
- 姓名
- 状态
- 最近登录时间
- 创建时间
- 更新时间

说明：

- OA 账号为一期唯一身份标识。
- 若 OA 回调账号未配置，默认拒绝访问，并记录失败原因。

### 8.3 用户组

字段：

- 用户组 ID
- 用户组名称
- 成员关系
- 创建时间
- 更新时间

### 8.4 授权策略

字段：

- 策略 ID
- 应用 ID
- 授权主体类型：用户或用户组
- 授权主体 ID
- 授权结果
- 有效期
- 创建时间
- 更新时间

说明：

- 一期只实现允许访问策略。
- 禁用、拒绝优先级等复杂策略留到后续版本。

### 8.5 ZT 会话

MySQL 表建议：

```text
zt_sessions
```

字段：

- token
- 用户 ID
- OA 账号
- 应用 ID 列表或授权缓存版本
- 创建时间
- 过期时间
- 最近访问时间
- 失效时间
- 失效原因

默认有效期：

- 8 小时，可通过配置调整。

说明：

- `token` 应保存摘要值，不建议明文保存原始 session token。
- `/auth/check` 查询会话时必须同时校验未过期、未失效、用户状态有效。
- 过期会话通过定时清理任务删除或归档。

### 8.6 OA 回调 ticket

MySQL 表建议：

```text
oa_login_tickets
```

字段：

- ticket
- ticket 摘要
- OA 账号
- 创建时间
- 过期时间
- 使用状态
- 使用时间
- 使用来源 IP

默认有效期：

- 1 到 5 分钟。

说明：

- ticket 必须一次性使用。
- 校验成功后立即标记为已使用。
- 重复使用、过期使用和账号不匹配必须拒绝，并写入登录/回调日志。

### 8.7 Nonce 防重放

MySQL 表建议：

```text
oa_callback_nonces
```

字段：

- nonce
- OA 账号
- 来源 IP
- 创建时间
- 过期时间

说明：

- 回调校验成功或失败后，nonce 均应写入 MySQL。
- 有效期与回调时间窗口一致。
- 重复 nonce 直接拒绝。

### 8.8 登录失败计数

MySQL 表建议：

```text
login_fail_counters
```

字段：

- OA 账号
- 来源 IP
- 失败次数
- 统计窗口开始时间
- 最近失败时间
- 解锁时间

说明：

- 一期可先只记录失败次数和来源，不强制锁定。
- 如需要防爆破，可按账号和 IP 维度设置短时间失败阈值。

### 8.9 审计日志

访问日志、登录日志和管理操作日志写入 MySQL。

保留策略：

- 默认保留 90 天。
- 超期日志、过期会话、过期 ticket、过期 nonce 通过定时清理任务删除或归档。

## 9. 网关接入设计

### 9.1 OpenResty 鉴权

网关对 OA 请求执行 `auth_request`：

```text
location / {
  auth_request /_ztrust_auth_check;
  proxy_pass http://oa_backend;
}
```

内部鉴权 location：

```text
location = /_ztrust_auth_check {
  internal;
  proxy_pass http://ztrust_control_plane/auth/check;
  proxy_pass_request_body off;
  proxy_set_header Content-Length "";
  proxy_set_header Cookie $http_cookie;
  proxy_set_header X-Original-Host $host;
  proxy_set_header X-Original-URI $request_uri;
  proxy_set_header X-Real-IP $remote_addr;
  proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
}
```

### 9.2 未登录跳转

当 `/auth/check` 返回 401 时，网关应将浏览器跳转到：

```text
/auth/oa/start?return_to=<original_url>
```

`return_to` 必须进行 URL 编码，并由控制面再次校验。

### 9.3 访问日志上报

网关在请求结束后通过 `log_by_lua` 上报访问元数据。

要求：

- 审计上报超时时间应较短，建议 100ms 到 300ms。
- 上报失败不阻断用户访问。
- 失败需进入网关本地错误日志，便于排查。

## 10. 部署、灰度与回滚

### 10.1 部署组件

一期部署组件：

- OpenResty / Nginx 网关
- Go 控制面服务
- MySQL
- 管理后台静态资源，随 Go 服务内嵌或由网关托管

### 10.2 灰度方式

推荐方式：

- 先使用 OA 测试环境验证。
- 再通过测试域名或少量用户 hosts/DNS 指向零信任网关。
- 灰度用户范围先选择 IT 或关键用户代表。
- 灰度周期不少于 1 到 2 周。

### 10.3 回滚方式

回滚目标：

- 10 分钟内恢复到原 OA 访问链路。

回滚手段：

- DNS/hosts 指回原 OA 入口。
- 网关配置切回直连 OA。
- 保留原 OA 登录入口和原访问链路。

### 10.4 配置要求

必须配置：

- OA 访问域名
- OA 后端地址
- OA 登录页地址
- OA 回调密钥
- ZT Cookie 名称和 Domain
- MySQL DSN
- 日志保留天数

## 11. 测试与验收标准

### 11.1 认证测试

- 未登录访问 OA 时跳转 OA 登录页。
- OA 登录成功后稳定回到零信任回调地址。
- 回调成功后浏览器获得 `ztrust_session`。
- ticket 过期、重复使用、签名错误、nonce 重放均被拒绝。
- `return_to` 非法域名或开放跳转尝试被拒绝。

### 11.2 授权测试

- 未授权用户访问 OA 被拦截。
- 授权用户可正常访问 OA。
- 用户组授权生效。
- 策略变更后权限及时生效。
- 应用停用后所有用户无法通过零信任入口访问。

### 11.3 OA 兼容性测试

- OA 首页访问正常。
- OA 登录流程正常。
- Cookie / Session 不丢失。
- 页面跳转正常。
- 审批流程正常。
- 附件上传下载正常。
- 大文件请求不异常中断。
- PC 浏览器主流版本兼容。

### 11.4 审计测试

- 每次访问生成访问日志。
- 日志包含账号、IP、路径、状态码、耗时、UA。
- 登录成功和失败均有日志。
- 管理后台操作有审计记录。
- 可按账号、IP、时间、路径、状态码查询。

### 11.5 灰度验收

- 灰度用户连续稳定使用不少于 1 到 2 周。
- 核心 OA 流程无阻断性问题。
- 出现问题可在 10 分钟内回滚。
- 网关错误率、延迟和资源占用在可接受范围内。
- 日志可用于安全排查和访问溯源。

## 12. 风险与后续演进

### 12.1 主要风险

- OA 回调改造不到位会影响认证闭环安全性。
- OA 代理兼容性是一期最大实施风险，必须先测试环境验证。
- 单机网关会成为关键访问节点，必须保留原链路回滚方案。
- Cookie Domain、SameSite、HTTPS 配置不当会导致登录态丢失。
- 日志包含路径、UA、IP 等敏感元数据，需要限制后台访问权限。
- 策略误配置可能导致用户无法访问 OA，必须记录操作日志。

### 12.2 默认处理策略

- OA 登录口自身加固、WAF、防爆破和漏洞修复继续保留。
- 所有 OA 回调必须验签、防重放、防过期。
- 管理操作必须写入审计日志。
- 请求体和响应体仅按规则记录，保存前脱敏，默认截断，查看动作写入管理操作日志。
- 管理后台仅允许内网或管理网段访问。
- 生产变更必须先灰度再扩大范围。

### 12.3 后续演进

二期可考虑：

- LDAP / AD / SSO 对接。
- MFA。
- 异常访问告警。
- 访问统计报表。
- 双机高可用。
- Redis 会话与授权缓存。
- OpenSearch / ClickHouse / 统一日志平台接入。
- 更多 Web 应用入口接入。
- 移动端浏览器和 OA App 兼容验证。
- 更细粒度路径授权。
